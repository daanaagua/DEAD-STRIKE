# DEAD STRIKE 第二轮游戏库扩展设计

## 1. 目标

本轮在不重做首页结构的前提下，为 `dead-strike.com` 新增 `22` 个可索引、可内链、可直接游玩的小游戏详情页，扩大站点对以下搜索意图的覆盖：

- 僵尸射击与僵尸生存
- 战术 / 军事 FPS
- 多人 / 竞技 / Arena 枪战
- 狙击与精准射击
- 防守型 / 守点型生存射击

本轮目标不是单纯增加页面数量，而是在保持站点主题仍然围绕“射击、FPS、生存、僵尸”主簇的前提下，补足目前较弱的长尾词簇，并让新页面通过目录页、分类页、详情页横向推荐和站点地图进入稳定的抓取路径。

## 2. 当前站点上下文

截至本设计编写时，站点已经具备以下基础：

- 首页 `index.html` 仍以 `DEAD STRIKE` 为核心主玩页。
- 已有共享游戏元数据文件 `game-library.js`，其中维护现有游戏信息、分类页数据和推荐池。
- 已有三大分类页：
  - `/games/zombie-games/`
  - `/games/fps-games/`
  - `/games/shooter-games/`
- 现有游戏页总量约三十余个，主题已经覆盖：
  - 普通射击页较多
  - 僵尸类页面较多
  - 军事和生存类页面已有一定基础
- 当前相对薄弱的词簇包括：
  - 多人竞技射击
  - 狙击类页面
  - battle royale / 大地图战区意图
  - 防守型 / 守点型生存射击

因此，本轮不适合继续大量补入与现有页面高度相似的普通僵尸 FPS 页面，否则容易导致页面互相抢词、文案难以拉开差异、分类页主题变得模糊。

## 3. 本轮确定的产品决策

以下内容已经确认：

- 扩库范围：选择相对更宽的近邻题材，不只限于纯僵尸 FPS。
- 页面数量：本轮目标新增 `22` 页。
- 来源策略：以已有已验证来源为主；如有更贴合主题且可免费嵌入的新来源，可以少量纳入。
- 首页策略：本轮不做明显改版，不重做首页门户结构。
- 实施重点：新页本身、分类页承接、目录页承接、详情页横向推荐、`sitemap.xml` 更新。

## 4. 本轮扩库的主题边界

### 4.1 应当纳入的主题

本轮新增页面应优先从以下主题中筛选：

- 战术 / 军事 FPS
- 现代枪战 / 特种兵任务
- 多人 / 团队 / Arena 射击
- 狙击 / 精准射击
- 僵尸防守 / 生存守点 / 僵尸围攻
- 大地图战区 / battle royale 相邻意图
- 仍然属于射击主簇的科幻 / 堡垒 / 地图对战页面

### 4.2 不应纳入的主题

以下主题不在本轮范围内，即使可嵌入也应排除：

- 赛车
- 纯跑酷
- 纯休闲点击
- 纯平台跳跃
- 纯解谜
- 与射击 / 枪战 / 生存战斗弱相关的游戏

### 4.3 主题控制原则

- 新页必须仍然服务于“browser shooter / FPS / zombie / survival / sniper / tactical combat”这一总体语义圈。
- 不为了凑够数量而引入明显偏题的页面。
- 如果候选游戏虽然是动作类，但无法自然落到射击或战斗意图中，则不采用。

## 5. 22 个新页面的结构分配

本轮建议按五个组别分配 `22` 个新页面：

### 5.1 战术 / 军事 FPS：6 页

目标词示例：

- tactical shooter
- commando game
- special ops game
- counter terrorist game
- modern war shooter
- gun strike online

目的：

- 扩充当前军事枪战与现代战区类意图
- 减少站点对“僵尸”单一主题的依赖
- 让 FPS 分类页更像真正的第一人称枪战入口，而非僵尸专题的附页

### 5.2 多人 / 竞技 / Arena 枪战：5 页

目标词示例：

- multiplayer shooter
- online gun game
- pvp shooter
- team deathmatch browser game
- arena gun battle

目的：

- 补足当前多人 / 竞技类页面数量不足的问题
- 增强 `shooter-games` 与 `fps-games` 两个分类页的差异化承接能力
- 为详情页之间提供更清晰的横向推荐簇

### 5.3 狙击 / 精准射击：4 页

目标词示例：

- sniper game online
- sniper mission
- city sniper shooter
- elite sniper game

目的：

- 强化目前较弱的狙击簇
- 提高页面标题和描述的可区分度
- 让现有的 `sniper-master`、`counter-craft-sniper`、`sniper-clash-3d`、`lone-wolf-strike` 等页拥有更完整的相邻网络

### 5.4 僵尸变体 / 守点 / 生存防守：4 页

目标词示例：

- zombie defense game
- zombie siege
- survival defense shooter
- base defense zombie game

目的：

- 继续保持站点在僵尸主簇中的强度
- 但重点转向“防守型”、“守线型”、“阵地型”变体，避免再堆普通僵尸 FPS

### 5.5 近邻长尾补位：3 页

候选方向示例：

- battle royale 邻近词
- blocky shooter / voxel shooter
- sci-fi shooter / space combat shooter

目的：

- 在不偏离主簇的情况下扩大语义边界
- 给目录页和分类页提供新的组合路径
- 测试更多长尾词是否具备收录与点击空间

## 6. 详情页模板约束

本轮新增详情页继续沿用现有详情页结构，不对整站视觉框架做重构。每个新页应具备：

- 独立路由 `/<slug>/index.html`
- 唯一的 `<title>`
- 唯一的 `<meta name="description">`
- 正确的 canonical
- 可加载的游戏 iframe
- 一段独立的开场导语
- 一段玩法或节奏说明
- 一段“为什么值得玩 / 适合谁玩”说明
- 一组 6 个邻近推荐链接

### 6.1 页面文案差异化要求

每个新页的文案必须至少在以下四个维度中体现出明显差异：

- 主打法：
  - sniper
  - arena
  - tactical
  - survival defense
  - zombie siege
- 节奏：
  - slow precision
  - fast rush
  - wave hold
  - team firefight
- 场景：
  - city
  - bunker
  - subway
  - rooftop
  - fort
  - lab
  - voxel world
- 搜索意图：
  - 每页只承接一个明确主词
  - 副词不超过两个

### 6.2 标题与描述控制规则

- 不允许多个页面使用几乎相同的标题骨架，只改一个词。
- 不允许多个页面的描述只是把游戏名替换后复用同一句模板。
- 如果两个候选游戏的搜索意图无法拉开，应只保留更强者。

## 7. 候选游戏筛选方法

本轮不应先凑够 22 个再强行落页，而应先建立一个 `30-36` 个候选页的候选池，再筛选出最终上线的 `22` 个。

筛选顺序如下：

### 7.1 第一关：可嵌入

候选页必须满足：

- 可在 iframe 中稳定加载
- 无登录墙
- 无明显报错或空白壳
- 无明显禁止嵌入情况

### 7.2 第二关：题材匹配

候选页必须明确属于以下方向之一：

- tactical shooter
- FPS
- sniper
- arena shooter
- zombie shooter
- survival defense
- multiplayer shooter

不满足则淘汰。

### 7.3 第三关：与现有页面的差异化

若候选页与现有页面或同批候选页高度相似，则只保留更强的一页。判断标准包括：

- 标题意图是否几乎相同
- 玩法节奏是否几乎相同
- 场景包装是否几乎相同
- 关键词是否会明显互抢

### 7.4 第四关：缩略图与命名质量

优先保留以下页面：

- 缩略图清晰
- 游戏名可自然形成 SEO 标题
- 页面命名不冗长、不乱码、不难以解释

### 7.5 第五关：来源稳健性

- 已有已验证来源优先
- 新来源仅作补位
- 新来源数量不应过多，以免影响整体稳定性

## 8. 分类页与目录页承接设计

既然首页本轮不做明显改版，新页能否被发现和抓取，主要取决于以下承接层。

### 8.1 `/games/` 总目录页

目录页承担“新入库总入口”的角色。本轮应：

- 扩展现有新页展示区
- 将新增页分成两层展示：
  - 核心新增：8 页
  - 更多新页：14 页
- 每张卡片保留不超过 2 个标签，避免信息噪音

目录页的目标不是塞满全部文案，而是为新页提供稳定、集中、可抓取的第一层入口。

### 8.2 `/games/zombie-games/`

该页只承接明确僵尸意图页面，适合纳入：

- 僵尸射击
- 僵尸生存
- 僵尸防守
- 僵尸围攻

不应因为某个页面“也有枪”就全部纳入。

### 8.3 `/games/fps-games/`

该页只承接第一人称体验明确的页面，适合纳入：

- 第一人称僵尸 FPS
- 第一人称战术射击
- 第一人称多人枪战
- 第一人称狙击页面（如果核心体验仍为 FPS）

### 8.4 `/games/shooter-games/`

该页承接 broader shooter 页面，包括：

- 竞技枪战
- arena shooter
- 狙击
- 非僵尸战术射击
- 其他动作射击补位页

### 8.5 分类挂载规则

- 每个新页至少进入 1 个分类页
- 每个新页最多进入 2 个分类页
- 禁止一个页面同时进入全部 3 个分类页

这样可以保持分类主题清楚，避免分类页失去边界。

## 9. 详情页之间的横向推荐设计

每个新详情页需要在现有详情页模板内保留 `6` 个横向推荐链接。

推荐优先级固定如下：

1. 同子簇页面：3 个
2. 同大簇页面：2 个
3. 回流核心页或强势 evergreen 页：1 个

回流页优先选择：

- `/`
- 或站内该簇最强的现有 evergreen 页面

### 9.1 横向推荐的目的

- 避免新页成为孤岛
- 形成专题簇，而不是分散孤立页
- 给搜索引擎提供稳定的同主题关系图谱

### 9.2 横向推荐的限制

- 不允许明显无关的页面为了凑数硬插入推荐区
- 推荐逻辑优先考虑主题相关性，而不是单纯用程序补满

## 10. sitemap 与抓取承接要求

本轮新增页面必须全部写入 `sitemap.xml`，并满足以下条件：

- `games/` 总目录页的更新时间同步刷新
- 有变更的分类页更新时间同步刷新
- 最终上线的 slug 与 `sitemap.xml` 中的 URL 一一对应
- 不允许保留占位 URL
- 不允许将未上线或已淘汰候选页写入 sitemap

## 11. 上线质量门槛

在允许 push 前，本轮需要满足以下最低质量要求。

### 11.1 数量要求

- 最终新增页面数量为 `22`

### 11.2 内链要求

每个新页至少具备以下三种稳定入链：

- 来自 `/games/`
- 来自至少 1 个分类页
- 来自至少 1 个详情页推荐区

### 11.3 页面质量要求

每个新页必须通过以下检查：

- 标题唯一
- 描述唯一
- canonical 正确
- iframe 可玩
- 页面无明显空区块
- 缩略图可正常显示
- 无 404 内链

### 11.4 抽样截图要求

至少检查以下页面：

- `/games/`
- `/games/zombie-games/`
- `/games/fps-games/`
- `/games/shooter-games/`
- 至少 4 个新详情页

截图的目的不是验证视觉改版，而是确认：

- 新卡片已经出现
- 链接结构正确
- 重点区块可见
- 没有明显布局断裂

## 12. 非目标

本轮明确不做以下事项：

- 不重做首页主结构
- 不把首页改成全站门户首页
- 不引入框架或构建系统
- 不重构整站视觉语言
- 不扩展到与射击弱相关的题材

## 13. 预期结果

如果本轮按本设计执行，站点应得到以下结果：

- 新增 `22` 个明确可索引的新游戏详情页
- 站点从“以僵尸 / FPS 为主”进一步扩展到“战术、竞技、狙击、防守型射击”的近邻词簇
- 不依赖首页重做，也能通过目录页、分类页、详情页横链和 sitemap 形成稳定抓取网络
- 新页之间的相似度被控制在可接受范围内，降低关键词互抢风险
- 后续如要继续扩库，可直接沿用本轮的候选筛选与分类挂载规则
